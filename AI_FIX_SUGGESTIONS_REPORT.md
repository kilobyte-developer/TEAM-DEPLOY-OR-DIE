# AI Fix Suggestions Report

## Architecture

AI fix suggestions are generated during source-code test generation after potential logic issues are detected. The flow is additive:

1. Source code is uploaded and analyzed as before.
2. Semantic suites are generated by Gemini source-code generation or the local fallback.
3. Each `potentialLogicIssues` item is enriched by a dedicated Gemini fix-suggestion call.
4. The enriched issue object is returned to the UI and stored in Supabase through the existing `semantic_test_cases.potential_logic_issues_json` field.

No upload, execution, coverage, dashboard, historical record, HTML report, or database schema contract was changed.

## Prompt Strategy

The fix prompt sends Gemini:

- Full uploaded Python source.
- Function or method name.
- Detected logic issue message.
- Existing confidence signal.

Gemini must return strict JSON with:

- `issueSummary`
- `currentCode`
- `suggestedCode`
- `explanation`
- `confidence`
- `severity`
- `potentialImpact`

The prompt explicitly asks for reasoning from source code and semantic intent, not deterministic replacement.

## Failure Handling

Fix suggestions use only `GEMINI_AI_FIXATIONS_API_KEY`.

If the key is missing, Gemini is unavailable, or the response is invalid, the system keeps test generation working and attaches:

`AI Fix Suggestion Unavailable`

Execution, coverage, report download, Supabase storage, and dashboard analytics continue.

## Storage Strategy

The project requirement says not to modify the database schema. The existing JSON column is used:

`semantic_test_cases.potential_logic_issues_json`

Each logic issue can now include a nested `fixSuggestion` object and metadata such as severity, confidence, provider, model, and timestamp. Older historical records with only `{ message, confidence }` remain compatible.

## Examples

Detected issue:

`Function name suggests checking even numbers but implementation appears to return True for odd numbers.`

AI fix suggestion:

- Severity: `High`
- Confidence: `95`
- Current: `return n % 2 == 1`
- Suggested: `return n % 2 == 0`
- Explanation: The function name indicates even-number validation, while the current condition returns true for odd values.
- Potential Impact: Could cause incorrect parity validation in downstream business rules.
